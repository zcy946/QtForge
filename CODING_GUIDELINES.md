# 编码与命名风格

本文档约定**命名**与**注释**写法，便于在多人协作或 AI 辅助编码时保持一致。与具体仓库冲突时，以**当前文件、当前模块的既有风格**为准。

---

## 1. 命名

### 1.1 通用 C++

- **类型（类、结构体、枚举）**：`PascalCase`。项目内可对自有类型使用统一**前缀**（例如 `Xlc`），与第三方库区分。
- **函数、方法**：`camelCase`；公开 API 建议动词开头、语义完整。
- **成员变量**：前缀 **`m_`** + `camelCase`（如 `m_data`、`m_maxLineCount`）。
- **局部变量、参数**：`camelCase`；短生命周期可用简短名称，但避免含糊的 `tmp`、`ret` 滥用。
- **宏、全文件常量**：习惯上全大写或项目统一前缀；与所依赖库（Qt、spdlog 等）的惯例冲突时，**以库为准**。
- **命名空间、文件名**：与模块、主类名一致；`.cpp` 内可用**匿名命名空间**封装翻译单元私有实现。

### 1.2 Qt 信号与槽（在使用 Qt 时）

- **声明**：可使用 Qt 宏 **`Q_SIGNALS:`**、**`public Q_SLOTS:`**、**`private Q_SLOTS:`**（与项目其余文件保持一致即可）。
- **发射**：使用 **`Q_EMIT`**。
- **信号名**：前缀 **`sig_`** + 语义，例如 `sig_configLoaded`、`sig_filterRuleChanged`。
- **槽名**：前缀 **`slot_`**；若槽对应**某一对象上的某一信号**，可采用 **`slot_<对象语义>_<信号语义>`**，例如 `slot_searchEdit_textChanged`、`slot_configManager_configLoaded`（对应信号 `sig_configLoaded` 时，对象语义写在中间）。
- 通过字符串调用槽（如 **`QMetaObject::invokeMethod`**）时，名字须与声明**完全一致**。

## 2. 注释与文档

### 2.1 Doxygen（公开 API、头文件）

对需要被阅读或生成的声明，使用块注释，常用标签：

| 标签 | 用途 |
|------|------|
| `@brief` | 一句话说明职责或效果 |
| `@param` | 各参数含义 |
| `@return` | 返回值；无返回值可省略 |
| `@note` | 前置条件、线程、与相关 API 的关系、注意事项 |

代码中的类型、函数名在注释里可用反引号标出，例如 `` `MyClass::init` ``、`` `Qt::UserRole` ``，便于阅读。

**示例：**

```cpp
/**
 * @brief 使用默认构建配置完成一次性初始化。
 * @return `true` 表示关键资源已就绪；`false` 表示降级路径或仅部分功能可用。
 * @note 重复调用行为由实现定义；若提供重载，在注释中说明彼此关系。
 */
static bool init();
```

文件头是否包含 `@file`、`@author`、`@date` 等，与**同目录其它头文件**对齐即可。

### 2.2 长实现文件中的分段注释（与 Doxygen 互补）

在**较长的 `.cpp`** 中，可用**非 Doxygen** 的等号/横线注释划分逻辑块，便于扫读；**不替代**对外 API 上的 Doxygen。

- **一级分段**（大块）：

```cpp
/* ==================== 分段名称 ==================== */
```

- **二级分段**（子块）：

```cpp
/* ------------------ 子项名称 ------------------ */
```

将 `分段名称`、`子项名称` 换成真实语义；同一文件内建议 **`===` 表大块、`---` 表其下子块**，层级清晰即可。

---

## 3. 与命名、注释相关的实现习惯

- **修改已有文件时**：新代码的命名与注释风格尽量与**该文件现有写法**一致；避免同文件内混用多种成员前缀或注释格式。
